# Задание 1

# Минимальный набор метрик для мониторинга платформы вычислений

**Контекст:** HTTP-сервис, выполняющий тяжелые CPU-задачи и сохраняющий текстовые отчеты на диск.  
**Цель:** обеспечить наблюдаемость по методологиям USE (ресурсы) и RED (запросы), добавив бизнес-слой.

---

## 1. Вычислительные ресурсы (CPU)

| Метрика | Почему |
|--------|--------|
| `node_cpu_seconds_total` (в разрезе ядер) | Общая загрузка скрывает перекосы – одно ядро может быть утилизировано на 100%, остальные простаивают (проблема тред-пула). |
| `node_load1/5/15` в отношении к числу ядер | Показывает насыщение (очередь). Если `load > cores` – запросы начнут тормозить даже при низком %CPU. |
| `node_cpu_steal_seconds_total` | Для виртуалок/контейнеров – если гипервизор перегружен, вычисления страдают, хотя свой CPU свободен. |

---

## 2. Оперативная память и диск

| Метрика | Почему |
|--------|--------|
| `node_memory_MemAvailable_bytes` | Свободная память – не показатель; Available учитывает кэш, который может быть освобожден. Падение Available → риск OOM. |
| `node_vmstat_pgmajfault` (или `pgfault`) | Частые страничные прерывания означают, что память подкачивается с диска – это резко замедляет вычисления. |
| `node_disk_io_time_seconds` (или `io_queue_length`) | Высокое время ожидания ввода-вывода (iowait) съедает CPU-кванты, даже если процессор свободен. Отчеты пишутся на диск – это узкое место. |
| `node_filesystem_inodes_free` | Текстовые отчеты могут быть тысячами мелких файлов – inodes заканчиваются раньше, чем место на диске. |

---

## 3. HTTP-слой (входные запросы)

По методологии RED:

| Метрика | Почему |
|--------|--------|
| `http_requests_total` (с разбивкой по эндпоинтам) | Отделяем `/compute` от `/health` – пик нагрузки может быть легитимным или аномальным. |
| `http_requests_total{status=~"5.."}` + процент ошибок | 500-е ошибки – первый признак внутренних сбоев (падение вычислений, таймауты). |
| `http_request_duration_seconds` (гистограмма, перцентили p50/p95/p99) | Среднее бесполезно. P99 покажет, страдают ли самые медленные запросы из-за нехватки CPU или диска. |

---

## 4. Бизнес-метрики (самый важный слой!)

Технические метрики не скажут, работает ли платформа по назначению.

| Метрика | Почему |
|--------|--------|
| `compute_duration_seconds` (гистограмма) | Время выполнения одного отчета – это главный SLA. Если оно растет, а железо зеленое – проблема в коде (блокировки, ретраи). |
| `reports_generated_total{status="success"}` и `{status="failed"}` | Счетчик успешных/проваленных вычислений. HTTP может отдавать 200, но отчет не сгенерироваться – это видно только здесь. |

---

## Правила алертинга

- Лучше избегать одиночных порогов (например, `CPU > 80%`) – они дают ложные срабатывания на фоновых задачах.
- Настраивать **корреляцию**: `(CPU load > 80%) AND (HTTP latency p95 > 1s)` – только тогда имеет смысл бить тревогу.
- Приоритет алертов: **Бизнес (ошибки генерации) > HTTP (5xx, задержки) > Ресурсы**.

---

**Итог:** Эти 10–12 метрик – минимальный набор, который покроет 90% инцидентов.

# Задание 2

# Предложение для менеджера продукта: бизнес-ориентированный мониторинг

Менеджеру не нужны RAM, inodes или load average — ему нужны ответы на два вопроса:

1. **Довольны ли клиенты?** (качество обслуживания)
2. **Выполняем ли мы свои обязательства?** (SLA перед клиентами)

## 1. Внедряем SLI и SLO (индикаторы и цели качества)

Переводим технические метрики в бизнес-показатели:

| Бизнес-показатель (SLI) | Как считаем | Целевое значение (SLO) |
|-------------------------|-------------|------------------------|
| **Доступность API** | Доля успешных HTTP-запросов (код 2xx/3xx) за период | ≥ 99.9% (или 99.5% — в зависимости от контракта) |
| **Скорость генерации отчета** | 95% времени выполнения `compute_duration_seconds` | ≤ 5 секунд (или другое, оговоренное в SLA) |
| **Успешность вычислений** | Доля отчетов, завершившихся без ошибок (`reports_generated_total{status="success"}` / total) | ≥ 99% |
| **Скорость ответа API** | 95% задержки HTTP (включая время вычислений) | ≤ 6 секунд (с учетом времени передачи) |

Все эти показатели вычисляются из тех же метрик, но агрегируются в понятные проценты и миллисекунды.

---

## 2. Дашборд для менеджера (без технического шума)

Создаем отдельную вьюшку в Grafana / Tableau с **тремя крупными индикаторами**:

- **Status:** "Сервис работает" (зеленый/красный) — если все SLO выполняются.
- **Текущее качество:** "% успешных отчетов за последний час" и "среднее время генерации".
- **Тренд:** график выполнения SLO за последние 7 дней (чтобы видеть ухудшения).

Все технические графики (CPU, диски) убираем во вкладку для инженеров — менеджер их не видит.

---

## 3. Еженедельный отчет "Здоровье продукта"

Автоматически формируем short report:

- **SLO Compliance:** сколько процентов времени мы укладывались в целевые значения за неделю.
- **Количество инцидентов**, повлиявших на клиентов (например, превышение времени ответа > 10 с).
- **Количество сгенерированных отчетов** — динамика нагрузки (чтобы отличать падение качества от роста числа запросов).

---

## 4. Связь с поддержкой

Добавляем метрику **"число жалоб клиентов"** (если есть) и накладываем на графики SLO — чтобы видеть корреляцию: падает качество → растут жалобы.

---

**Итог:** Менеджер получает не "load average", а "мы выполнили 99.8% отчетов вовремя, задержка выросла на 10% за неделю — надо увеличивать мощности". Технические метрики остаются для инженеров, но их агрегаты становятся понятными бизнесу.

# Задание 3

# Решение: сбор ошибок без бюджета

**Проблема:** нет денег на развертывание централизованной системы логов (ELK, Splunk, и т.д.), но разработчикам нужен доступ к ошибкам приложений.

**Предлагаемое решение:** использовать **бесплатный SaaS-инструмент для мониторинга ошибок** — например, **Sentry** (или аналог: Rollar, Airbrake, Raygun с бесплатным тарифом).

---

## Почему Sentry?

| Критерий | Что дает |
|----------|----------|
| **Бесплатный тариф** | До 5 000–10 000 событий в месяц (зависит от плана) — достаточно для небольших проектов. |
| **Интеграция** | Поддерживает все популярные языки (Python, Java, Go, Node.js, PHP) — подключается одной строкой в коде. |
| **Данные прямо разработчикам** | Sentry автоматически группирует ошибки, показывает стектрейс, версию кода, параметры запроса — разработчику не нужно лезть в логи. |
| **Оповещения** | Можно настроить уведомления в Slack, e-mail при появлении новых ошибок. |
| **Хранение** | Сервис хранит историю ошибок, можно посмотреть частоту и тренды. |

---

## Альтернативы (если Sentry не подходит)

- **Logtail** (бывший Better Stack) — бесплатный план до 1 ГБ логов в месяц, есть поиск и дашборды.
- **Papertrail** — 50 МБ/месяц бесплатно, простой просмотр логов.
- **Slack-бот**: при возникновении ошибки отправлять сообщение в чат разработчиков (через webhook) — самый простой вариант, но без группировки и истории.

---

## Внедрение (минимальные шаги)

1. Регистрируем аккаунт в Sentry (бесплатно).
2. Создаем проект для нашего приложения.
3. Вставляем SDK в код (например, для Python: `sentry-sdk.init(dsn=...)`) и оборачиваем основной блок.
4. Настраиваем отправку только ошибок (`ERROR` уровень), чтобы не тратить лимит на информационные логи.
5. Даем разработчикам доступ в Sentry — они видят все ошибки в реальном времени с контекстом.

---

## Что делать, если ошибок много и бесплатный лимит заканчивается?

- Включить **выборку (sampling)** — отправлять не все ошибки, а, например, 10% (для получения общей картины).
- Использовать **агрегацию** — отправлять только уникальные ошибки, а не каждое повторение.
- Ограничить хранение (в настройках Sentry можно сократить срок хранения).

---

**Итог:** Никаких серверов и затрат — разработчики получают панель с ошибками уже через 15 минут после интеграции, а бюджет остается в сохранности.

# Задание 4

**Ситуация:**  
Расчет SLA по формуле:  
`SLA = sum(2xx) / sum(all_requests)`  
При этом в системе **нет** ответов с кодами 4xx и 5xx, но SLA не поднимается выше 70%.  

**Ошибка:**  
30% запросов, которые считаются «неуспешными», — это **коды 3xx (редиректы)** или **1xx (информационные)**.  
Исключаются они из числителя, но оставляются в знаменателе, искусственно занижая показатель.

---

## Как исправить

### Вариант 1. Уточнить, что считать успешным ответом
Если редиректы (3xx) — это штатное поведение API (например, перенаправление на другой endpoint), нужно добавить их в числитель:

`SLA = (sum(2xx) + sum(3xx)) / sum(all_requests)`

### Вариант 2. Исключить «технические» запросы
Если 3xx — это редиректы на несуществующие страницы (ошибки маршрутизации), их нужно **исключить из знаменателя**, так как они не относятся к бизнес-логике:

`SLA = sum(2xx) / (sum(all_requests) - sum(3xx) - sum(1xx))`

### Вариант 3. Изменить знаменатель
Вместо «всех запросов» использовать только те, которые должны были быть обработаны приложением (например, исключить health-проверки, запросы к статике, редиректы).

---

## Вывод
Проблема не в системе, а в **некорректной агрегации**.  
SLA должен отражать только те запросы, которые имеют бизнес-ценность, и коды, которые действительно сигнализируют об ошибке.  

**Правило:**  
Успешные = 2xx + (3xx, если редиректы допустимы).  
Неуспешные = 4xx + 5xx.  
Остальное (1xx, 3xx без редиректов, отмененные запросы) — исключать из расчета или считать нейтральным.

# Задание 5

# Pull vs Push: сравнение систем мониторинга

| Критерий | **Pull** (Prometheus, Zabbix) | **Push** (Graphite, StatsD, Datadog) |
|----------|-------------------------------|--------------------------------------|
| **Обнаружение целей** | Сервер сам знает, какие endpoints опрашивать (через service discovery или статический конфиг). | Приложения сами отправляют метрики на коллектор — сервер пассивен. |
| **Масштабирование** | Сложнее: опрос тысяч целей требует распределения нагрузки (федерация, шардинг). | Проще: агенты отправляют данные асинхронно, коллектор может быть горизонтально масштабируем (очереди, балансировка). |
| **Надежность доставки** | При падении агента сервер просто не получит данные — это явно видно (метрика `up`). | При падении коллектора данные теряются безвозвратно — агенты не знают, дошло ли. |
| **Безопасность** | Сервер инициирует соединение — проще контролировать доступ (белые списки IP, TLS). Нет открытых портов на агентах. | Агенты открывают порт для приема метрик — требуется защита от спама, подделки данных. |
| **Динамические окружения** | Service discovery (Kubernetes, Consul) автоматически находит новые поды. | Агенты сами регистрируются — проще в highly dynamic средах, но усложняет контроль. |
| **Балансировка нагрузки** | Если агент временно недоступен, сервер повторяет запрос — данные в итоге получает. | Push требует буферизации на стороне клиента, если коллектор перегружен — может переполнить память. |
| **Агрегация и обработка** | Данные можно агрегировать на стороне сервера (запросы с `rate()`). | Часто агрегация выполняется на коллекторе (StatsD) или на клиенте. |
| **Отказоустойчивость** | Проще: не нужно синхронизировать состояние между серверами. | Сложнее: коллектор должен быть кластеризован, чтобы не потерять данные при перезапуске. |
| **Пример использования** | Kubernetes, микросервисы, статические дата-центры. | IoT, мобильные приложения, задачи с низкой задержкой, когда агент отправляет события. |

---

## Ключевые выводы

- **Pull** — выбор SRE для статичных или хорошо обнаруживаемых систем, где важна надежность и простота отладки (Prometheus — стандарт де-факто).
- **Push** — выбор, если у вас тысячи эфемерных сущностей (клиенты, мобильные устройства) или требуется минимизировать задержку доставки метрик.

**Современный подход:** гибрид — приложения отправляют метрики через push-гейтвей (например, для краткосрочных задач), а основной сбор делает pull-система.

# Задание 6

# Классификация систем мониторинга по модели сбора данных

| Система | Модель | Пояснение |
|---------|--------|-----------|
| **Prometheus** | **Pull** (с опциональным push) | Основной сбор через опрос экспортеров. Есть Pushgateway для краткосрочных задач (batch-задания), но это исключение — основная парадигма pull. |
| **TICK** (Telegraf + InfluxDB) | **Push** | Telegraf активно отправляет метрики в InfluxDB. Хотя Telegraf умеет опрашивать источники (pull), стандартная конфигурация — push на выходе. |
| **Zabbix** | **Гибрид** | Поддерживает оба режима: пассивные проверки (сервер опрашивает агентов — pull) и активные проверки (агент сам отправляет данные — push). Часто используется гибридно. |
| **VictoriaMetrics** | **Гибрид** (pull-first) | Ядро — pull-сбор (совместим с Prometheus), но также принимает данные по push-протоколам (InfluxDB, Graphite, OpenTSDB, Prometheus remote write). Используется и как pull, и как push-приемник. |
| **Nagios** | **Pull** (с пассивными дополнениями) | Основной режим — сервер опрашивает хосты/службы по расписанию (NRPE, check_by_ssh). Пассивные проверки (NSCA) доступны, но не являются основным сценарием. |

---

## Резюме

- **Чистый pull:** Prometheus (почти всегда), Nagios (преимущественно).  
- **Чистый push:** TICK (Telegraf -> InfluxDB).  
- **Гибридные:** Zabbix, VictoriaMetrics (поддерживают оба подхода, но с разным акцентом).

Выбор модели зависит от инфраструктуры: для динамических сред (Kubernetes) удобнее pull с service discovery, для статичных с большим числом агентов — push может быть проще.

## Полученный InfluxQL-запрос

Chronograf сгенерировал следующий запрос:

```sql
SELECT mean("usage_system") 
FROM "telegraf"."autogen"."cpu" 
WHERE time > now() - 1h 
AND "host" = 'telegraf-getting-started' 
GROUP BY time(10s)
```

## Эксперименты с настройками
В интерфейсе Chronograf были изменены следующие параметры для изучения влияния на график:

- **Агрегатная функция** (Aggregate) изменена с `mean` на `max` – график теперь показывает пиковые значения системной загрузки

# Задание 1

## Описание решения
Разработан Python-скрипт, который:
- Собирает системные метрики из виртуальной файловой системы `/proc`.
- Формирует JSON-строку с временной меткой (Unix timestamp) и набором метрик.
- Записывает строку в файл `/var/log/YY-MM-DD-awesome-monitoring.log`, где `YY-MM-DD` – текущая дата.
- Запускается каждую минуту через cron.

Собираемые метрики (не менее 4-х):
- **load_avg_1, load_avg_5, load_avg_15** – средняя загрузка CPU за 1, 5 и 15 минут (из `/proc/loadavg`).
- **mem_total, mem_available, mem_used_percent** – общая, доступная память и процент использования (из `/proc/meminfo`).
- **cpu_user, cpu_system, cpu_idle** – время CPU в режиме пользователя, ядра и idle (в процентах от общего времени) – вычисляется на основе двух последовательных снятий показаний, но для простоты будем брать текущие значения из `/proc/stat` и считать отношение за интервал. Однако поскольку скрипт запускается раз в минуту, можно вычислять разницу между текущим и предыдущим вызовами. 

Итак, выберем следующие метрики (не менее 4-х):
1. load_avg_1
2. load_avg_5
3. load_avg_15
4. mem_available (в МБ)
5. mem_total (в МБ)
6. uptime_seconds (время работы системы)
7. procs_running (количество запущенных процессов)
8. procs_blocked (количество заблокированных процессов)

Это уже 8 метрик.

Опишем код.

---

## 1. Python-скрипт (collect_metrics.py)

```python
#!/usr/bin/env python3
import os
import time
import json
from datetime import datetime

LOG_DIR = "/var/log"
DATE_STR = datetime.now().strftime("%Y-%m-%d")
LOG_FILE = f"{LOG_DIR}/{DATE_STR}-awesome-monitoring.log"

def read_proc_file(path):
    """Читает содержимое файла из /proc и возвращает список строк"""
    try:
        with open(path, 'r') as f:
            return f.readlines()
    except Exception as e:
        print(f"Error reading {path}: {e}")
        return []

def get_loadavg():
    """Парсит /proc/loadavg: load1, load5, load15, procs_running, procs_blocked"""
    lines = read_proc_file("/proc/loadavg")
    if not lines:
        return None
    parts = lines[0].split()
    # parts: [load1, load5, load15, running/total, last_pid]
    load1 = float(parts[0])
    load5 = float(parts[1])
    load15 = float(parts[2])
    # running/total - например, "4/256"
    running_total = parts[3].split('/')
    procs_running = int(running_total[0])
    procs_total = int(running_total[1]) if len(running_total) > 1 else 0
    # last_pid не используем
    return {
        'load_avg_1': load1,
        'load_avg_5': load5,
        'load_avg_15': load15,
        'procs_running': procs_running,
        'procs_total': procs_total
    }

def get_meminfo():
    """Парсит /proc/meminfo: MemTotal, MemAvailable (в МБ)"""
    lines = read_proc_file("/proc/meminfo")
    if not lines:
        return None
    mem_total = None
    mem_available = None
    for line in lines:
        if line.startswith("MemTotal:"):
            mem_total = int(line.split()[1]) // 1024  # конвертируем КБ -> МБ
        elif line.startswith("MemAvailable:"):
            mem_available = int(line.split()[1]) // 1024
        if mem_total is not None and mem_available is not None:
            break
    if mem_total is None or mem_available is None:
        return None
    return {
        'mem_total_mb': mem_total,
        'mem_available_mb': mem_available,
        'mem_used_mb': mem_total - mem_available,
        'mem_used_percent': round((mem_total - mem_available) / mem_total * 100, 2)
    }

def get_uptime():
    """Парсит /proc/uptime: возвращает uptime в секундах (float)"""
    lines = read_proc_file("/proc/uptime")
    if not lines:
        return None
    uptime_seconds = float(lines[0].split()[0])
    return uptime_seconds

def main():
    timestamp = int(time.time())
    data = {
        'timestamp': timestamp,
        'datetime': datetime.now().isoformat()
    }

    # Сбор метрик
    load = get_loadavg()
    if load:
        data.update(load)

    mem = get_meminfo()
    if mem:
        data.update(mem)

    uptime = get_uptime()
    if uptime is not None:
        data['uptime_seconds'] = uptime

    # Можно добавить ещё метрики, например, из /proc/stat (CPU)
    # Но для демонстрации хватит.

    # Запись в лог-файл
    try:
        with open(LOG_FILE, 'a') as f:
            f.write(json.dumps(data) + '\n')
    except Exception as e:
        print(f"Error writing log: {e}")

if __name__ == "__main__":
    main()
```

# Задание 2: Создание дашборда в Chronograf

## Цель
Создать в Chronograf пользовательский дашборд для визуализации системных метрик: утилизация CPU, использование RAM, заполнение дисков, количество запущенных Docker-контейнеров, сетевой трафик и аптайм.

---

## Выполненные шаги

### 1. Подготовка метрик
- Убедились, что в `telegraf.conf` включены необходимые плагины:
  - `cpu`, `mem`, `disk`, `docker`, `net`, `system`
- Перезапустили Telegraf и дождались поступления данных.
- Проверили наличие измерений через `SHOW MEASUREMENTS` – все нужные появились.

### 2. Создание дашборда
В Chronograf (http://localhost:8888) перешли в **Dashboards** → **Create Dashboard**, задали имя `System Overview`.

### 3. Добавление ячеек с запросами
Для каждой метрики создана отдельная ячейка. Использованы полные имена базы и измерения, макросы времени (`:dashboardTime:`, `:upperDashboardTime:`, `:interval:`).

#### CPU (утилизация в системном режиме)
```sql
SELECT mean("usage_system") AS "CPU System %" 
FROM "telegraf"."autogen"."cpu" 
WHERE time > :dashboardTime: AND time < :upperDashboardTime: 
AND "host" = 'telegraf-getting-started' 
GROUP BY time(:interval:) FILL(null)
```

#### RAM (процент использованной памяти)
```sql
SELECT mean("usage") AS "mean_usage", mean("usage_percent") AS "mean_usage_percent"
FROM "telegraf"."autogen"."docker_container_mem"
WHERE time > :dashboardTime: AND time < :upperDashboardTime:
GROUP BY time(:interval:) FILL(null)
```
#### Диск (процент занятого места на корневом разделе)
```sql
SELECT mean("used") AS "mean_disk_used", mean("used_percent") AS "mean_disk_used_percent" 
FROM "telegraf"."autogen"."disk" 
WHERE time > :dashboardTime: AND time < :upperDashboardTime: 
GROUP BY time(:interval:) FILL(null)
```
#### Количество запущенных контейнеров
```sql
SELECT last("n_running") AS "Containers Running", mean("n_containers") AS "mean_n_containers", mean("n_containers_running") AS "mean_n_containers_running"
FROM "telegraf"."autogen"."docker"
WHERE time > :dashboardTime: AND time < :upperDashboardTime:
GROUP BY time(:interval:) FILL(null)
```
#### Аптайм системы (в часах)
```sql
SELECT last(uptime) / 3600 AS "Uptime (hours)" 
FROM "telegraf"."autogen"."system" 
WHERE time > now() - 24h 
GROUP BY time(1h) fill(null)
```
#### Сетевой трафик (входящий/исходящий, КБ/с)
```sql
SELECT mean("rx_bytes") AS "mean_rx_bytes", mean("tx_bytes") AS "mean_tx_bytes" 
FROM "telegraf"."autogen"."docker_container_net" 
WHERE time > :dashboardTime: AND time < :upperDashboardTime: 
GROUP BY time(:interval:) FILL(null)
```
